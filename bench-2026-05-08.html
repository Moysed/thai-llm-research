Bench Report · 2026-05-08

Six Thai LLMs benchmarked on the actual Enggenius pronunciation prompt

50 samples per provider against the actual PROMPT_TEMPLATE from GeminiPronunciationService.cs. Latency, schema reliability, score sanity, and Thai output completeness — what shipping looks like, not what the marketing page says.

Verdict

Ship

Stay on Gemini Flash-Lite for now. Student says a sentence → feedback shows up in 2.1s typical, 4.8s on a bad day. Quick enough to keep flow. Add retry on 503/timeout — 4% of calls failed transiently in this run, which means roughly 1 in 25 students would see a broken response without a fallback.

→ see the numbers

Don't ship

Hosted Typhoon 2.5-30b-a3b is not viable for Enggenius. Median wait 6.2s, bad-day wait 10.2s — 2× the latency budget. Picture a student saying "The weather is beautiful today" and watching a spinner for 10 seconds before any feedback appears. They've already opened TikTok. Schema is rock-solid (50/50) and Thai quality is richer than Gemini, but the latency wall is real.

→ see the numbers

Watch

Schema completeness gap. Gemini returns valid JSON but its words array is just word names — no per-word score, no per-word Thai feedback. Typhoon ignores the schema and returns full word_scores with bilingual feedback per word. The bench rates both as "schema-valid"; Enggenius's actual student-facing UX needs the per-word feedback. Tighten the schema before declaring a winner.

→ see what the student sees

Round 2 finding

THaLLE-0.2-ThaiLLM-8B-fa (KBTG) is the surprise. Sovereign 8B model on Thai gov GPUs — p50 3.7s, schema-clean 10/10 on the smoke run. Faster than the hosted 30b Typhoon, faster than Gemini Flash-Lite, and produces richer per-word bilingual feedback than either. Caveat: only n=10/provider so far; a full 5-iter rerun is needed before betting prod on it.

At a glance

Reading the numbers

  • p50 = median. Half of requests finish faster than this. The "typical" experience.
  • p95 = 95% of requests finish faster than this. The "bad day" experience that 1 in 20 students will hit.
  • p99 = the worst 1%. The "I'm gonna refresh and try again" experience.
  • Schema-fail = the response was valid JSON but missing required fields (no overall_score, no recognized_text) — bench can't grade the student's answer.
  • Smoke run = a small first pass (1 iteration here, n=10) to confirm everything works before committing to a long full run.
  • Sovereign = runs on Thai government compute (ThaiSC supercomputer), free public access, no data leaves Thailand.

For context: 5 seconds is roughly two breaths — about as long as a student will wait before losing focus. 10 seconds is enough to switch tabs to TikTok.

What was tested

Prompt: verbatim PROMPT_TEMPLATE + LOCALE_INSTRUCTION from GeminiPronunciationService.cs on Enggenius main. The same prompt that ships to students.

Cases (10 representative student responses):

  • perfect_match — student nails the reference
  • minor_error — typo-level mispronunciation ("coffe" vs "coffee")
  • wrong_word — semantic swap ("cook" for "book")
  • missing_word — dropped word
  • thai_response — answered in Thai instead of English
  • silent — empty input
  • nonsense — "blah blah"
  • code_switch — Thai-English mix ("I will go to the ประชุม")
  • long_phrase — Thai-accented full sentence
  • single_word — common mispronunciation ("pronounciation")

Providers:

  • Typhoon typhoon-v2.5-30b-a3b-instruct via api.opentyphoon.ai — note: typhoon-v2.1-12b-instruct from prior research notes is no longer served, only the 30b is currently live on the hosted endpoint.
  • Gemini gemini-2.5-flash-lite via generativelanguage.googleapis.com with responseSchema enforced.

Settings: temperature=0.3, max_tokens=2500, response_format=json_object. Sequential calls, no concurrency, single client machine.

The schema completeness gap

This is the part the headline numbers miss. Both providers passed the bench's schema_valid check. The actual outputs are different products.

What the student actually sees:

  • With Gemini: ✓ ✓ ✓ ✓ ✗ — five marks next to five words. No coaching. The student knows they got "cook" wrong, but not what to do about it.
  • With Typhoon: ✓ ✓ ✓ ✓ ✗ + "ฝึกออกเสียง 'book' ด้วยเสียงยาว 'oo' และเสียง 'k' ที่นุ่มนวล" per word, in Thai. The student knows what to do.

Same JSON validity check passes for both. They are not the same product.

Gemini — schema-valid, but empty

Gemini · case: wrong_word · words array

[ {"word": "She"}, {"word": "is"}, {"word": "reading"}, {"word": "a"}, {"word": "cook"} ]

No score field. No feedback field. No tip on how to fix the "cook" → "book" error.

Why: responseSchema in Gemini's API forces strict adherence to the bench's TEXT_SCHEMA. That schema declares score as a property but not as required, and doesn't declare feedback at all. Gemini drops both. Typhoon, given only response_format=json_object (no schema), follows the natural-language prompt and returns the rich object.

Fix path: tighten TEXT_SCHEMA to require { word, score, feedback: { en, th } } per word — then re-run the bench. Until then, the latency comparison is fair, but the quality comparison favors Typhoon by default because it's writing essays while Gemini is filling out a form.

Action

If Enggenius uses Gemini in prod without retry-on-503 logic, ~4% of pronunciation grades break. Either add a LiteLLM gateway with provider fallback, or wrap the Gemini call in 2× retry with exponential backoff for 503/timeout.

What this means for Enggenius

  1. Don't migrate off Gemini yet. The latency win is too big to give up. Hosted Typhoon 30b is a non-starter for the real-time grading flow.
  2. Add retry/fallback on Gemini. 4% transient failure rate is unacceptable for a student-facing product. LiteLLM gateway with Gemini → Typhoon failover is the cheapest fix and keeps Typhoon in play as a backstop where p95 latency can spike.
  3. Tighten the JSON schema before any future bench. Make per-word score and feedback.{en,th} required. Right now Gemini's "schema-valid" output would be a degraded UX in prod.
  4. Re-evaluate Typhoon when self-hosted. The 30b on hosted infra is too slow — Typhoon-S-ThaiLLM-8B (sovereign Thai gov compute, free, natively OpenAI-compatible — see Round 2) or self-hosted Typhoon 2.1-12B Q4_K_M on a Vast L40 should hit <3s p95 if served via vLLM. That's the real comparison.
  5. Code-switching is Typhoon's actual edge. 85–89 vs Gemini's flat 70 on "I will go to the ประชุม" — if Enggenius students mix Thai filler words while practicing English (likely), Typhoon's grading is more forgiving and arguably more correct. Worth a follow-up bench focused only on code-switch cases.

Round 2 — Sovereign ThaiLLM Playground (8B models)

Sample size caveat

Preliminary, n=10/provider (1 iteration × 10 cases). The full 5-iteration run was started but stalled past 25 minutes — the sovereign endpoint shows variable tail latency under sequential load and the run was cancelled. Numbers below are directional; do not quote as final p95.

Reliability — schema compliance is the real story

All 4 sovereign models stayed online (zero HTTP errors, zero timeouts in 40 calls). The differentiation is in JSON contract adherence:

  • THaLLE 10/10 — every response had valid JSON with required recognized_text and overall_score. The only sovereign model that earns the JSON-mode label.
  • OpenThaiGPT 10/10 — schema-clean across all 10 cases. Slower than THaLLE but equally reliable.
  • Typhoon-S 6/10 — 4 schema fails on edge cases (silent, code_switch, long_phrase, wrong_word). Manual retest succeeded — appears to be variance/truncation under max_tokens=2500, may need 3500+.
  • Pathumma 3/10 — failed schema on 7/10 cases. Reads more like a model alignment issue than a bench artifact: it's a "think" reasoning model (Qwen3-think-3.0.0) and seems to invent its own JSON shape ignoring the prompt's required fields. overall_score is the most common missing field on edge inputs.
Disqualified for prod

Pathumma-ThaiLLM-qwen3-8b-think-3.0.0 is not viable for Enggenius as the JSON-grading endpoint. It's a reasoning model, not a structured-output model — even when the JSON validates, it gets the actual grading wrong.

The failure cascade in plain terms: a student says "cook" instead of "book". Pathumma gives them 87.5/100 and tells them their problem was the word "reading". The student walks away thinking they did well, never knowing the actual mistake — and now they "know" their "reading" is wrong, which it isn't. The student loses trust in the app AND learns the wrong lesson. Worst of both worlds.

Updated implications for Enggenius

  1. Run a full 5-iter bench against THaLLE and OpenThaiGPT before any decision. The smoke is too small to commit to. But if THaLLE holds at p50 ~3.7s with 50/50 schema across 50 samples, it becomes the new default candidate — free, sovereign, faster than Gemini, and produces full bilingual per-word feedback by default.
  2. Pathumma is out. Reasoning models are not the right architecture for structured grading. NECTEC's "think" lineage prioritizes chain-of-thought over schema adherence — wrong shape for this product.
  3. Typhoon-S deserves a second look with bigger max_tokens. Schema fails appear to be truncation. Bump to 3500 and rerun before disqualifying — it's already the closest to "full Thai-aware Typhoon" available without paying.
  4. The 12B Typhoon variant (recommended in the May research write-up) is unreachable. Hosted 30b is too slow; the 8B-S sovereign sibling is reachable. Self-hosting the 12B Q4_K_M on Vast L40 is now the only way to test that specific size point.
  5. Sovereign models actually deliver per-word bilingual feedback by default — without needing strict response schemas. THaLLE and OpenThaiGPT both filled word_scores as a dict (word→score) plus grammar_feedback / content_feedback / explanation objects with {en, th}. The schema-completeness gap that made Gemini look weak in Round 1 doesn't exist for them.

Caveats

  • Single client, single network, BKK residential. Latency for users in other regions will differ — Gemini's POPs vs opentyphoon.ai's TH-only routing means Typhoon may close the gap for non-TH users (and widen it for TH users).
  • No concurrency. Real Enggenius traffic is likely 1-10 concurrent students; if rate limits or queueing kicks in, both providers' p95 will rise.
  • typhoon-v2.1-12b-instruct (the smaller model recommended in the original Thai LLM research write-up) is no longer served on opentyphoon.ai. Only the 30b is currently live. The 8B variants are reachable via the new sovereign ThaiLLM Playground (login required) — adding that endpoint to the bench is the obvious next step.
  • Schema completeness was not enforced in this bench. Both providers passed the loose validator; only manual inspection of results.json revealed the Gemini empty-words issue.
  • Round 2 sample size is small (n=10/provider). The full 5-iter run was cancelled when sequential calls to the sovereign endpoint stalled past 25 minutes — likely queueing / cold-start variance during peak hours on the shared ThaiSC GPU pool. Re-run during off-hours or with backoff between calls.
  • Sovereign endpoint rate limit is 5 req/s, 200 req/min. The bench's sequential pacing stays well under, but a concurrent prod workload would saturate within ~40 students grading simultaneously — sovereign is for SME, not consumer scale.
