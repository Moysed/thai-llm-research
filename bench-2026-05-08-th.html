สรุปการตัดสินใจ

Ship ได้

ใช้ Gemini Flash-Lite ต่อไปก่อน — p95 4.8 วินาที อยู่ใต้ขอบเขต 5 วินาทีสำหรับ UX แบบ real-time ได้สบายๆ เพิ่ม retry logic เมื่อเจอ 503/timeout เพราะรอบนี้ 4% ของ call ล้มเหลวแบบชั่วคราว

อย่า Ship

Typhoon 2.5-30b-a3b แบบ hosted ใช้กับ Enggenius ไม่ได้ — p95 10.2 วินาที เกินขอบเขต latency 2 เท่า ตรวจ Thai feedback ยาวๆ ทีไรก็ทะลุ 10 วินาทีตลอด schema นิ่งดีนะ (50/50) คุณภาพ Thai ก็ดี แต่กำแพง latency คือกำแพง

ระวัง

ช่องว่างความสมบูรณ์ของ schema Gemini คืน JSON ที่ valid ก็จริง แต่ array words มีแค่ชื่อคำ — ไม่มีคะแนนรายคำ ไม่มี feedback ภาษาไทยรายคำ ส่วน Typhoon ไม่สนใจ schema ตรงๆ แต่ส่ง word_scores เต็มพร้อม feedback สองภาษาต่อคำ bench ของเรามองทั้งคู่ว่า "schema-valid" — แต่ UX ที่นักเรียนเห็นจริงต้องการ feedback รายคำ ต้องเข้มงวด schema ก่อนตัดสินผู้ชนะ

รอบ 2: ของแถมที่ไม่คาดคิด

THaLLE-0.2-ThaiLLM-8B-fa (KBTG) คือม้ามืด — โมเดลอธิปไตย 8B รันบน GPU ของรัฐไทย p50 3.7 วินาที, schema สะอาด 10/10 จากการ smoke run เร็วกว่า Typhoon 30b แบบ hosted เร็วกว่า Gemini Flash-Lite และให้ feedback สองภาษารายคำที่ครบถ้วนกว่าทั้งคู่ ดูรอบ 2 ด้านล่าง ข้อแม้: ตอนนี้แค่ n=10 ต่อ provider ต้อง full 5-iter rerun ก่อนจะลง prod

ภาพรวม

สิ่งที่ทดสอบ

Prompt: ใช้ PROMPT_TEMPLATE + LOCALE_INSTRUCTION ตรงจาก GeminiPronunciationService.cs บน main ของ Enggenius — prompt เดียวกับที่ส่งให้นักเรียนจริงทุกวันนี้

10 cases ที่เป็นตัวแทนคำตอบนักเรียน:

  • perfect_match — นักเรียนพูดตรงเป๊ะ
  • minor_error — สะกดผิดเล็กน้อย ("coffe" แทน "coffee")
  • wrong_word — เปลี่ยนคำที่ความหมายต่าง ("cook" แทน "book")
  • missing_word — ตกคำ
  • thai_response — ตอบเป็นภาษาไทยแทนอังกฤษ
  • silent — ไม่พูด (input ว่าง)
  • nonsense — "blah blah"
  • code_switch — สลับไทย-อังกฤษ ("I will go to the ประชุม")
  • long_phrase — ประโยคยาวสำเนียงไทย
  • single_word — ออกเสียงผิดที่พบบ่อย ("pronounciation")

Providers:

  • Typhoon typhoon-v2.5-30b-a3b-instruct ผ่าน api.opentyphoon.ai — หมายเหตุ: typhoon-v2.1-12b-instruct ที่งานวิจัยเดิมแนะนำ ตอนนี้ไม่มีให้ใช้แล้ว เหลือแต่ 30b เท่านั้นบน hosted endpoint
  • Gemini gemini-2.5-flash-lite ผ่าน generativelanguage.googleapis.com โดยบังคับ responseSchema

การตั้งค่า: temperature=0.3, max_tokens=2500, response_format=json_object เรียกแบบ sequential ไม่มี concurrency จากเครื่อง client เดียว

Gemini — schema valid แต่กลวง

Gemini · case: wrong_word · words array

[ {"word": "She"}, {"word": "is"}, {"word": "reading"}, {"word": "a"}, {"word": "cook"} ]

ไม่มี score ไม่มี feedback ไม่มีคำแนะนำว่าจะแก้ "cook" → "book" ยังไง

เพราะอะไร: responseSchema ของ Gemini บังคับให้ตามรูปร่าง TEXT_SCHEMA ของ bench แบบเข้มงวด schema ของ bench ประกาศ score เป็น property แต่ไม่ได้ required และไม่ประกาศ feedback เลย Gemini ก็ทิ้งทั้งคู่ ส่วน Typhoon ที่มีแค่ response_format=json_object (ไม่มี schema) ทำตาม prompt ภาษาธรรมชาติเลยส่ง object เต็ม

ทางแก้: เข้มงวด TEXT_SCHEMA โดย required ทั้ง { word, score, feedback: { en, th } } ต่อคำ — แล้วรัน bench ใหม่ ตอนนี้การเปรียบเทียบ latency ยุติธรรมแล้ว แต่การเปรียบเทียบคุณภาพเอนเอียงเข้าหา Typhoon เพราะมันเขียนเรียงความขณะที่ Gemini แค่กรอกฟอร์ม

ต้องทำ

ถ้า Enggenius ใช้ Gemini ใน prod โดยไม่มี retry-on-503 ราว 4% ของการตรวจการออกเสียงจะพัง ทางแก้ที่ง่ายที่สุดคือใส่ LiteLLM gateway ที่ fallback ระหว่าง provider หรือห่อ Gemini call ด้วย retry 2 ครั้งแบบ exponential backoff สำหรับ 503/timeout

ความหมายต่อ Enggenius

  1. อย่าย้ายออกจาก Gemini ตอนนี้ — ที่ได้จาก latency มันใหญ่เกินจะทิ้ง Typhoon 30b แบบ hosted ไม่ผ่านสำหรับ flow ตรวจการออกเสียงแบบ real-time
  2. เพิ่ม retry/fallback บน Gemini — อัตรา failure ชั่วคราว 4% ยอมรับไม่ได้สำหรับ product ที่นักเรียนเห็น ทางที่ถูกที่สุดคือ LiteLLM gateway ที่ Gemini → Typhoon failover และเก็บ Typhoon เป็น backstop ตอน p95 latency เด้ง
  3. เข้มงวด JSON schema ก่อน bench รอบหน้า — บังคับ score และ feedback.{en,th} ต่อคำให้ required ตอนนี้ output ของ Gemini ที่ "schema-valid" จะเป็น UX ที่อ่อนใน prod
  4. ทบทวน Typhoon เมื่อ self-host — 30b บน hosted infra ติด bandwidth — Typhoon-S-ThaiLLM-8B (รันบน GPU ของรัฐไทย ฟรี OpenAI-compatible) หรือ self-host Typhoon 2.1-12B Q4_K_M บน Vast L40 น่าจะแตะ p95 ต่ำกว่า 3 วิได้ถ้า serve ผ่าน vLLM นั่นคือการเปรียบเทียบจริง
  5. Code-switching คือจุดแข็งจริงของ Typhoon — 85–89 เทียบกับ 70 คงที่ของ Gemini บน "I will go to the ประชุม" ถ้านักเรียน Enggenius สลับคำเชื่อมไทยตอนฝึกอังกฤษ (ซึ่งน่าจะใช่) Typhoon ตรวจอ่อนกว่าและเหมาะสมกว่า ควร bench ตามอีกครั้งเฉพาะ code-switch cases

รอบ 2 — ThaiLLM Playground (โมเดลอธิปไตย 8B)

ข้อแม้เรื่อง sample size

เบื้องต้นเท่านั้น n=10/provider (1 iteration × 10 cases) full 5-iter run เริ่มไปแล้วแต่ค้างเกิน 25 นาที — sovereign endpoint แสดงความผันผวน tail latency เมื่อโดน sequential load หนักๆ เลยยกเลิก ตัวเลขด้านล่างคือทิศทาง อย่าใช้อ้างอิง p95 จริง

ThaiLLM Playground ของรัฐไทยเปิดให้ใช้ 4 โมเดล 8B ที่ tune มาเฉพาะภาษาไทยจาก 4 ทีม ผ่าน API ที่ ฟรีและ OpenAI-compatible ที่ https://api.thaillm.or.th/v1 Rate limit: 5 req/s, 200 req/min Auth: Authorization: Bearer <key> (บทความของ somkiat.cc ที่บอกว่า "ไม่ compatible กับ OpenAI" ตกยุคแล้ว — API ตอนนี้ตามรูป OpenAI เต็มๆ)

ความน่าเชื่อถือ — ตัวชี้วัดจริงคือการตามสัญญา schema

ทั้ง 4 โมเดลอธิปไตยอยู่ออนไลน์ดี (ไม่มี HTTP error ไม่มี timeout ใน 40 calls) ความต่างอยู่ที่ การเคารพ JSON contract:

  • THaLLE 10/10 — ทุก response มี JSON ถูกต้องพร้อม recognized_text และ overall_score โมเดลอธิปไตยตัวเดียวที่สมศักดิ์ศรี JSON-mode
  • OpenThaiGPT 10/10 — schema สะอาดทุก case ช้ากว่า THaLLE แต่นิ่งเท่ากัน
  • Typhoon-S 6/10 — schema fail 4 ครั้งใน case ขอบเขต (silent, code_switch, long_phrase, wrong_word) retest แบบ manual แล้วผ่าน — น่าจะเป็น variance/truncation ที่ max_tokens=2500 อาจต้องเพิ่มเป็น 3500+
  • Pathumma 3/10 — schema fail 7/10 cases อ่านเหมือนปัญหาการ align ของโมเดลมากกว่า bench artifact — เป็น "think" reasoning model (Qwen3-think-3.0.0) เลยคิดสร้างรูป JSON ของตัวเองโดยข้าม required fields ของ prompt overall_score คือ field ที่หายไปบ่อยสุดบน edge inputs

ความหมายต่อ Enggenius ที่อัปเดต

  1. รัน full 5-iter bench กับ THaLLE และ OpenThaiGPT ก่อนตัดสินใจ — smoke น้อยเกินไปจะ commit ถ้า THaLLE ยืนได้ที่ p50 ~3.7 วินาทีพร้อม schema 50/50 ใน 50 samples มันจะกลายเป็น candidate default ตัวใหม่ — ฟรี อธิปไตย เร็วกว่า Gemini และให้ feedback สองภาษารายคำเต็มโดย default
  2. Pathumma ตัดออก reasoning model ไม่ใช่สถาปัตย์ที่ใช่สำหรับ structured grading สาย "think" ของ NECTEC ให้ความสำคัญ chain-of-thought มากกว่าการเคารพ schema — ผิดรูปสำหรับ product นี้
  3. Typhoon-S ควรลองอีกครั้งด้วย max_tokens ที่ใหญ่กว่า schema fail น่าจะเป็น truncation ดัน max เป็น 3500 แล้วรันใหม่ก่อนตัดสิทธิ์ — มันคือสิ่งที่ใกล้เคียง "Typhoon Thai-aware เต็มรูป" ที่สุดที่ใช้ได้โดยไม่ต้องจ่าย
  4. Typhoon 12B ตามที่งานวิจัย พฤษภาคม แนะนำ ใช้ไม่ได้แล้ว 30b แบบ hosted ช้าเกิน 8B-S อธิปไตยใช้ได้ self-host 12B Q4_K_M บน Vast L40 คือทางเดียวจะทดสอบ size point นั้นได้
  5. โมเดลอธิปไตยให้ feedback สองภาษารายคำมาเต็มอยู่แล้ว — โดยไม่ต้องบังคับ response schema THaLLE และ OpenThaiGPT เติม word_scores เป็น dict (word→score) พร้อม object grammar_feedback / content_feedback / explanation ที่มี {en, th} ครบ ช่องว่าง schema completeness ที่ทำให้ Gemini ดูอ่อนในรอบ 1 ไม่มีกับพวกนี้

ข้อแม้

  • เครื่อง client เดียว เครือข่ายเดียว ที่อยู่อาศัยกรุงเทพฯ Latency สำหรับผู้ใช้ภูมิภาคอื่นจะต่าง — Gemini มี POPs หลายที่ ส่วน opentyphoon.ai routing แต่ในไทย Typhoon อาจใกล้กันสำหรับผู้ใช้นอกไทย (และห่างกว่าสำหรับผู้ใช้ในไทย)
  • ไม่มี concurrency การจราจรจริงของ Enggenius น่าจะมี 1-10 นักเรียนพร้อมกัน ถ้าโดน rate limit หรือ queue ทั้งคู่จะ p95 สูงขึ้น
  • typhoon-v2.1-12b-instruct (ตัวเล็กที่งานวิจัย Thai LLM ฉบับเดิมแนะนำ) ไม่มีบน opentyphoon.ai แล้ว มีแต่ 30b ตอนนี้ ตัว 8B variants เข้าผ่าน ThaiLLM Playground ของรัฐ (ต้อง login) — เพิ่ม endpoint นั้นใน bench คือก้าวต่อไปที่ชัด
  • ความสมบูรณ์ของ schema ไม่ได้บังคับใน bench นี้ ทั้งคู่ผ่าน validator แบบหลวม ตรวจมือใน results.json เท่านั้นถึงเห็นปัญหา empty-words ของ Gemini
  • Sample size รอบ 2 น้อย (n=10/provider) full 5-iter run ถูกยกเลิกเมื่อ sequential calls ค้างเกิน 25 นาที — น่าจะเป็น queueing/cold-start variance ในช่วงพีคของ shared GPU pool ของ ThaiSC ลองรันช่วงนอก peak hour หรือใส่ backoff ระหว่าง call
  • Rate limit ของ sovereign endpoint คือ 5 req/s, 200 req/min การจัดจังหวะ sequential ของ bench อยู่ใต้ขอบเขตสบายๆ แต่ workload prod แบบ concurrent จะเต็มพอนักเรียนตรวจพร้อมกันราว 40 คน — sovereign สำหรับ SME ไม่ใช่ scale ระดับผู้บริโภค
