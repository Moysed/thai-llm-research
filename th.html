วิเคราะห์ Build-vs-Buy · ภาพรวม Thai LLM

แทน Gemini & GPT ในแอปเรียนภาษาอังกฤษด้วย Thai LLM

สรุปงานวิจัยเดือนพฤษภาคม 2026 ของ Thai LLM แบบ open-source — Typhoon, OpenThaiGPT, Pathumma, SEA-LION — สำหรับรับ workload จริง (ตรวจการออกเสียง + สร้างคำถาม) พร้อมต้นทุนของการ self-host, stack ที่แนะนำ และแผน rollout แบบแบ่งเฟส

📅 พฤษภาคม 2026 📦 Enggenius (Unity / WebGL) 🇹🇭 Thai-first ⏱️ อ่าน 18 นาที

สรุปสั้น

วิธีอ่าน

เอกสารด้านล่างเป็น default frame สำหรับ SME / single-product workload (50k–500k req/วัน). ถ้าอยู่ที่ enterprise หรือ government scale (~1,000+ CCU, ผู้ใช้หลายพัน, ข้อมูลที่กำกับโดยกฎหมาย) คำแนะนำเปลี่ยน — กระโดดไปที่ Enterprise & Government Deployment.

หมายเหตุเรื่อง audio mode: สำหรับ SME, hybrid (text → Thai LLM, audio → OpenAI) รับได้. แต่สำหรับข้อมูลที่กำกับ/รัฐ — audio ต้องย้ายมาที่ Whisper + Wav2Vec2 ใน local เพราะส่งเสียงผู้เยาว์ไป endpoint ในอเมริกาขัดกับ PDPA

ข้อสรุป (default สำหรับ SME)

อย่าเพิ่ง self-host ทั้งหมดในตอนนี้ — ทำทีละขั้นจะปลอดภัยและประหยัดเวลา dev กว่า:

  1. วันนี้: ติดตั้ง LiteLLM (proxy gateway สำหรับเรียก LLM ที่รองรับหลาย provider) คั่นระหว่างแอปกับ LLM. ตั้งให้เรียก Typhoon hosted API (opentyphoon.ai) เป็นหลัก, มี Gemini เป็น fallback. ผลคือ — rate-limit หายทันที, ค่า GPU เป็นศูนย์, ใช้เวลาตั้งค่า ~1 วัน
  2. เมื่อ traffic เกิน ~50k calls/day แบบต่อเนื่อง: ค่อย self-host Typhoon-12B บน GPU เช่า (ราคาคงที่ ~$220/เดือน). เปลี่ยนได้โดยแก้แค่ endpoint ใน LiteLLM, ไม่ต้องแก้โค้ดแอป

วิธีนี้ทำให้แอปไม่ผูกกับ vendor ใด vendor หนึ่ง — ถ้า Typhoon ล่ม Gemini รับช่วงต่อ, ถ้า Gemini แพงขึ้น สลับ vendor ได้โดยแก้ config ไม่กี่บรรทัด

ปัญหาปัจจุบัน

แอปเรียนภาษาอังกฤษสำหรับนักเรียนไทยตอนนี้ใช้ AI 2 เจ้าในการตรวจการออกเสียง:

  • Gemini 2.5 Flash-Lite (Google) — รับ transcript เป็นข้อความ → ให้คะแนนการออกเสียง (โหมด text)
  • GPT-4o-audio-preview (OpenAI) — รับเสียงพูดดิบๆ → ให้คะแนนพร้อมจับ accent, intonation (โหมด audio)

การพึ่ง vendor ภายนอกแบบนี้มี 3 ความเสี่ยงที่ควรกังวล:

  1. Vendor lock-in (ผูกขาดกับเจ้าเดียว). ถ้า Google ปรับราคา Gemini ขึ้น 3 เท่าเดือนหน้า เราควบคุมไม่ได้. ถ้า OpenAI deprecate gpt-4o-audio-preview (เคยทำกับ model อื่นมาแล้ว) ทุกการเรียก API จะพังทันทีจนกว่าจะ migrate. ที่แย่กว่านั้น — ถ้าใครคนใดคนหนึ่งล่มในชั่วโมงทอง โหลดเรียน เด็กรอ feedback ไม่ได้
  2. ต้นทุน token เพิ่มตรงตามจำนวนผู้ใช้. ที่ 1,000 user/วัน ค่าใช้จ่ายอาจอยู่ที่ ~$5/วัน ($150/เดือน). พอขยายเป็น 100,000 user/วัน กลายเป็น ~$500/วัน ($15,000/เดือน) — hosted API ไม่มีส่วนลดเชิงปริมาณที่จริงจัง. ยิ่ง user เยอะ ยิ่งเสียค่า token ตรงๆ
  3. Sovereignty (อธิปไตยของข้อมูล). เสียงพูดของนักเรียนไทยถูก process ที่ data center ในอเมริกา, อยู่นอก data residency ของไทย/อาเซียน. ลูกค้าระดับองค์กร โดยเฉพาะหน่วยราชการหรือโรงเรียนใหญ่ๆ บางแห่งไม่อนุญาตให้ส่งข้อมูลผู้ใช้ออกนอกประเทศ — ตัดโอกาสการขายไปเลย

เป้าหมาย: หาเส้นทางไปสู่ Thai-hosted LLM (ไม่ว่าจะ self-host เอง, ใช้ hosted API ของ opentyphoon.ai, หรือผ่าน managed Thai cloud provider) โดยที่ product ไม่พังในวันแรก และ migration ทำได้แบบค่อยเป็นค่อยไป. คำตอบที่ถูกต้องขึ้นกับ scale — ดูที่ "วิธีอ่าน" ด้านบน

Workload จริง (อย่าข้าม)

การเลือก LLM โดยไม่วัด workload ก่อนคือการเดา — เลือกผิดแล้วต้อง migrate ใหม่เสียเวลามากกว่ามานั่งวัดก่อน 1 วัน. มาดูว่า call แต่ละครั้งของแอปจริงๆ หน้าตาเป็นยังไง:

  • ขนาด prompt: input ประมาณ 500 tokens (รวม system prompt + ประโยคอ้างอิง + สิ่งที่นักเรียนพูด). 500 tokens ≈ ~1 หน้า A4 แบบย่อ — ไม่ใหญ่
  • Output: 200–400 tokens แบบ structured JSON (JSON ที่มีโครงสร้างตายตัว — เช่น ต้องมี field overall_score เป็นตัวเลข 0-100, words[] เป็น array ของคำ + คะแนนแต่ละคำ, ฯลฯ). โมเดลต้องตอบให้ตรง schemaเป๊ะๆ ไม่งั้นโค้ด parse ไม่ได้
  • Latency: real-time — นักเรียนกำลังรอ feedback อยู่. เป้า p95 ≤ 5 วินาที (95% ของ request ต้องเสร็จภายใน 5 วินาที). p50 (ค่ามัธยฐาน) ควรอยู่ที่ ~1-2 วินาที
  • ปริมาณ: ประมาณ 500–10,000 calls/วัน ตามจำนวนนักเรียนที่ active. ที่ peak (ช่วงเย็นหลังเลิกเรียน) อาจกระจุกเป็นต่อนาที 50-100 requests
  • 2 โหมดการทำงาน:
    • text-mode — รับ transcript (เสียงที่ถูก speech-to-text แปลงเป็น text แล้ว) → ให้คะแนน. ใช้ LLM อะไรก็ได้ที่ฉลาด
    • audio-mode — รับเสียงพูดดิบๆ → ให้คะแนน (โมเดลฟัง accent, fluency, intonation จากเสียงจริง). ตอนนี้ยังไม่มี open Thai LLM ตัวไหนรับ audio ได้โดยตรง
  • ไม่ได้ใช้ function calling — แอปแค่บอก LLM ว่า "ตอบเป็น JSON ตาม schema นี้" ผ่าน response_schema (Gemini) หรือ response_format: json_object (OpenAI). ไม่ใช่ tool-use ที่ซับซ้อน
ข้อจำกัดสำคัญ — โหมด audio

ถ้าจะคงโหมด audio ไว้ (เพราะมันใช้ acoustic signal วัดการออกเสียงได้แม่นกว่า), จะแทน 1:1 ด้วย Thai LLM ตัวเดียวไม่ได้ — ต้องเพิ่มขั้นตอน Whisper (โมเดล speech-to-text ของ OpenAI ที่ open-source) มา preprocess เสียงเป็น transcript ก่อน, แล้วค่อยส่งให้ Thai LLM ประเมิน.

ทางเลือกที่ใช้ได้:

  • Hybrid (แนะนำ) — text-mode → Thai LLM, audio-mode → OpenAI ต่อ. ลดความซับซ้อนของการ migrate
  • Full self-host — Whisper-large-v3 + Typhoon-12B รันคู่กันบน GPU เดียว. แม่นกว่าในแง่ language quality แต่อาจสูญ acoustic signal บางส่วน
  • Text-only — เลิกโหมด audio ไปเลย ใช้แค่ transcript. ง่ายสุด แต่คุณภาพการประเมินจะลด

3 ตัวเลือกที่ดีที่สุด

หมายเหตุ: การจัดอันดับด้านล่างสมมติเป็น SME workload (hosted API หรือ self-host แบบไม่ซับซ้อน). ที่ enterprise/gov scale, โมเดลเดิมแต่ช่องทางการเข้าถึงเปลี่ยน — อาจไปผ่าน managed Thai cloud provider (เช่น SIAM.AI, NTT Thailand) แทนเรียก hosted API ตรง. ดูใน Enterprise / รัฐ สำหรับ decision tree

1Typhoon 2.x — SCB-10X

ภาษาไทยดีที่สุด Function calling มี hosted API Apache-2.0 (ตัว 2.5)

พัฒนาโดยทีม SCB-10X ของธนาคารไทยพาณิชย์. มีหลายขนาดให้เลือกตามงาน:

  • 4B (base = Gemma3) — เล็กพอรันบน GPU เดียวได้ที่ load production
  • 12B (Gemma3, context 128K tokens) — ขนาดกลาง ใช้กับ context ยาวๆ ได้
  • 30B-A3B (Qwen3 MoE — Mixture of Experts) — ขนาดใหญ่แต่ active แค่ 3B ตอน inference จริง, ได้คุณภาพระดับ 30B แต่เร็วเท่า 3B
  • 70B (Llama 3.1) — ใหญ่สุด คุณภาพสูงสุด แต่ต้องใช้ GPU ราคาแพง

เป็น Thai LLM ตัวเดียวที่มีตัวเลข function calling แบบ BFCL (Berkeley Function Calling Leaderboard — เกณฑ์มาตรฐานวัดความสามารถในการเรียก tool ของ LLM) ตีพิมพ์ — ได้คะแนน 75% สำหรับภาษาไทย / 79% สำหรับภาษาอังกฤษ, ชนะ baseline ของ Qwen2.5 และ Llama3 ในหมวดเดียวกัน.

มี hosted API ที่ opentyphoon.ai ราคา ~$0.10 ต่อ 1 ล้าน tokens (ถูกกว่า Gemini Flash 2.5 ประมาณ 14 เท่า ตาม 2.5 release post). ขึ้นอันดับ 1 ของทุก leaderboard ภาษาไทยที่เปิดเผยต่อสาธารณะ.

ทำไมเหมาะ (สำหรับ SME deployment):

  • ตัว 4B รันบน GPU เช่า L40 ตัวเดียว ($220/เดือน) ได้สบายแม้ที่ load production
  • JSON schema เชื่อถือได้ (ตอบตรง schema ทุกครั้ง — สำคัญมากเพราะแอปต้อง parse JSON)
  • Feedback ภาษาไทยอ่านเป็นธรรมชาติ ไม่ใช่สำนวนแปลแข็งๆ
  • ถ้าไม่อยาก self-host เลย — hosted API ใช้ได้เลย ไม่ต้องเขียนโค้ดเพิ่ม รองรับ OpenAI-compatible format

สำหรับ enterprise/government deployment, ใช้โมเดลเดียวกันแต่เข้าผ่าน managed Thai cloud — ดู ด้านล่าง

2OpenThaiGPT — NECTEC / AIEAT

เก่ง reasoning Qwen license (เพดาน 100M MAU) มี playground ฟรี

พัฒนาโดย NECTEC (สำนักงานพัฒนาวิทยาศาสตร์และเทคโนโลยีแห่งชาติ) ร่วมกับ AIEAT (Artificial Intelligence Entrepreneur Association of Thailand). โครงการระดับชาติ.

เวอร์ชันล่าสุดคือ 1.6 + R1-32B (ออก มี.ค. 2025).

  • ตัว R1-32B เป็นรุ่น reasoning (ใช้ chain-of-thought ในการคิด — เหมือน DeepSeek-R1, OpenAI o1). ทำคะแนน AIME24-TH 56.7 (ข้อสอบคณิตศาสตร์โอลิมปิกเวอร์ชันแปลไทย) และ MATH500-TH 83.8 — ชนะ DeepSeek-R1 ขนาด 70B ทั้งที่ตัวเองเล็กกว่าครึ่ง
  • รองรับ tool-calling ที่ inherit มาจาก Qwen2.5 base
  • Context: 131K tokens input / 8K output

มี playground ฟรีที่ openthaigpt.aieat.or.th สำหรับลองเล่น แต่ไม่มี enterprise SLA — ใช้ใน production ต้องเตรียม fallback เอง.

ทำไมเหมาะ: ถ้าในอนาคตต้องการ reasoning ขั้นสูง — เช่น coaching แกรมม่าที่ซับซ้อน, บทสนทนายาวข้ามบทเรียน, อธิบายทำไมประโยคนี้ผิด — ตัว R1-32B เป็นเพดานสูงสุดของ Thai LLM ในด้าน reasoning ตอนนี้.

ข้อจำกัด: รอบ release ช้ากว่า SCB-10X — ตามจังหวะของโครงการที่รัฐสนับสนุน, feature ใหม่ๆ มัก backport มาช้า. License ของ Qwen ติดเพดาน 100M MAU (ผู้ใช้ active ต่อเดือน) — สำหรับ SME ไม่มีปัญหา แต่สำหรับ B2C ที่ scale ใหญ่ต้องระวัง.

3Pathumma-LLM — NECTEC

Apache-2.0 คุณภาพระดับกลาง หน่วยงานราชการสนับสนุน

พัฒนาโดย NECTEC (เจ้าเดียวกับ OpenThaiGPT แต่คนละทีม) — โปรเจกต์ของรัฐที่เน้นเรื่องเสรีของ licenseเป็นหลัก.

  • Pathumma-llm-text-1.0.0 — base คือ Qwen2.5-7B, ขนาด 7B, ctx 128K
  • Pathumma-vision-2.0.0-preview — เวอร์ชันรับภาพ (multimodal) กำลังพัฒนา
  • Pathumma-audio-1.0.0 — เวอร์ชันรับเสียง (อาจเป็นทางเลือกในอนาคตสำหรับ audio-mode ของ Enggenius)

คุณภาพภาษาไทยตามหลัง Typhoon และ OpenThaiGPT 1.5-14B พอสมควร. มีคน eval แบบ independent น้อย — ส่วนใหญ่เป็นรายงานของ NECTEC เอง. Host ฟรีที่ aiforthai.in.th สำหรับทดลอง.

ทำไมเหมาะ: เป็น Thai LLM ตัวเดียวที่ใช้ license แบบ Apache-2.0 — เสรีที่สุดในกลุ่มนี้, ไม่มีเพดาน MAU, ไม่มีเงื่อนไขเชิงพาณิชย์. ใช้ในงานราชการ/B2B องค์กรขนาดใหญ่ที่ procurement เข้มงวดได้สบาย โดยไม่ต้องอธิบายเงื่อนไขของ Gemma ToU หรือ Qwen license.

ข้อจำกัด: คุณภาพด้อยกว่า Typhoon จริงๆ — โดยเฉพาะภาษาไทยแบบเป็นธรรมชาติและความสามารถในการตอบ JSON schema. แนะนำเป็น Plan B สำหรับเก็บไว้เผื่อกรณีที่ license ของ Gemma หรือ Qwen กลายเป็นปัญหาในอนาคต.

ตัวอื่นๆ ที่ควรรู้จัก (แต่ไม่ใช่ตัวเลือกหลัก)

  • SEA-LION v3 / v3.5 (AI Singapore) — ดีที่สุดถ้าต้องรองรับหลายภาษาในเอเชียตะวันออกเฉียงใต้ (Bahasa, Vietnamese, Tamil ฯลฯ). แต่ถ้าใช้แค่ภาษาไทย Typhoon ทำได้ดีกว่าเพราะ tune มาเฉพาะ
  • WangchanLion / WangchanX (VISTEC สถาบันวิทยาสิริเมธี) — เน้นเป็น fine-tuning pipeline (เครื่องมือ + dataset สำหรับเอาไป fine-tune เอง) มากกว่าตัวโมเดลสำเร็จรูป. แข็งใน Thai MRC (Machine Reading Comprehension — งานอ่านแล้วตอบคำถาม) แต่ในการสนทนาทั่วไปยังอ่อนกว่า Typhoon
  • THaLLE (KBTG ธนาคารกสิกร) — fine-tune เฉพาะทางสำหรับ การเงิน. เก่งเรื่อง CFA, การวิเคราะห์งบ. เกินความจำเป็นสำหรับแอปเรียนภาษา
  • SeaLLM-v3 ดั้งเดิม (Alibaba DAMO) — เคยเป็นตัวเลือกหลักสำหรับภาษาเอเชียตะวันออกเฉียงใต้ แต่ตอนนี้กลายเป็น legacy แล้ว — AI Singapore แยกออกมาทำ SEA-LION ต่อยอดเอง ไม่อัปเดตอีก

เศรษฐศาสตร์ของการ self-host

ตามจริงคือ: ที่ volume ระดับ SME (~50k req/day), การ self-host ไม่ได้ถูกกว่า hosted Gemini Flash-Lite เลย. ทำไม?

  • Hosted API คิดเงินตามการใช้จริง — เรียก 1 ครั้งจ่าย 1 ครั้ง. ที่ volume ต่ำ ค่าใช้จ่ายจึงต่ำมาก
  • Self-host มีต้นทุนคงที่ — เช่า GPU ไว้ตลอด 24 ชม. ต่อให้ไม่มีคนใช้ก็ต้องจ่าย. กำไรของการ self-host จะเริ่มเห็นชัดตอน volume สูงขึ้นจนเริ่ม "เต็ม" GPU

เพราะฉะนั้น การลงทุน self-host ในช่วงต้นคือการเสียเงินเพิ่ม เพื่อแลกกับสิ่งที่ไม่ใช่ราคา — เช่น ไม่ติด rate-limit, ควบคุม latency เอง, ข้อมูลไม่ออกประเทศ.

จุดคุ้มทุนของ self-host: เมื่อ traffic ผ่าน ~50k req/day อย่างต่อเนื่อง — ราคา fixed ของ GPU ($220) เริ่มถูกกว่า hosted API. ที่ 500k+ req/day, self-host ถูกกว่า hosted API หลายเท่า.

ตัวเลือก GPU เช่า (ตามราคา)

  • ถูกสุด — Vast.ai L40 40GB ~$0.31/ชม. (~$220/เดือน). รัน Typhoon 2.1-Gemma3-12B แบบ Q4_K_M (โมเดลถูก quantize ให้ขนาดเล็กลง) ได้สบาย. เหมาะสำหรับ Phase 3 ของแผน
  • ระดับกลาง — RunPod A100 80GB ~$0.79/ชม. ($350–570/เดือน ขึ้นกับเช่าแบบ on-demand หรือ reserved). รัน 12B แบบ FP16 (precision เต็ม) ได้พร้อมเหลือ memory สำหรับ KV-cache (cache ของ context ที่ผ่านมา ทำให้ตอบเร็วขึ้นในบทสนทนายาว)
  • Production — H100 80GB ~$1.50/ชม. ($1,100/เดือน). เกินความจำเป็นจนกว่าแอปจะมี traffic >200 requests/second (ระดับ enterprise). Typhoon 2.5-30B-A3B (MoE) เคลมว่าทำได้ 3,000 tokens/วินาทีที่ H100 — เร็วมาก
ข้อแนะนำ

อย่าลงทุน GPU ก่อน traffic จริงๆ จะถึงจุดคุ้มทุน. เริ่มจาก Typhoon hosted API ($10-25/เดือน) ก่อน ค่อยขยับขึ้น L40 ตอนเห็นเลข traffic ที่จริงในระบบ.

Stack สำหรับ production

มี 2 ส่วนสำคัญที่ต้องเลือก: (1) Routing layer — ตัวกลางระหว่างแอปกับ LLM provider, (2) Serving layer — ตัวรัน LLM จริงๆ (เฉพาะตอน self-host).

1. Routing layer: LiteLLM

LiteLLM คือ proxy server ที่ใส่คั่นระหว่างแอปกับ LLM. หน้าที่ของมัน:

  • รับ request ที่หน้าตาเหมือน OpenAI API (มาตรฐานที่ทุกคนคุ้น) แล้วส่งต่อไป provider ที่กำหนด — Typhoon, Gemini, Claude, OpenAI ฯลฯ
  • Fallback chain — ถ้า provider หลักล่ม สลับไปตัวสำรองอัตโนมัติ ไม่ต้องเขียน try/catch ในแอป
  • Retry + cooldown — ถ้า request fail ลองใหม่อัตโนมัติ มีหน่วงเวลาเพื่อไม่ spam
  • Budget cap — กำหนดเพดานค่าใช้จ่ายต่อเดือน เกินแล้ว block
  • Semantic cache — ถ้า prompt ใกล้เคียงกับที่เคยถามแล้ว (เช่น แบบฝึกหัดเดิม) ตอบจาก cache ทันที — ประหยัด token มหาศาล

Run ด้วย Docker container เดียว ฟรี. ทางเลือกอื่น:

  • OpenRouter — เป็น hosted SaaS (ไม่ต้อง deploy เอง) แต่ควบคุมได้น้อยกว่า, มี markup เล็กน้อย
  • Portkey — ระดับ enterprise (มี advanced guardrails, governance) แต่เกินความจำเป็นสำหรับ SME

ตัวอย่าง config:

# litellm-config.yaml — ตัวอย่าง
model_list:
  - model_name: pronounce            # ชื่อที่แอปเรียก
    litellm_params:
      model: openai/typhoon-v2.1-12b-instruct
      api_base: https://api.opentyphoon.ai/v1
      api_key: os.environ/TYPHOON_API_KEY

router_settings:
  # ถ้า Typhoon ล่ม → fallback ไป Gemini → ถ้า Gemini ล่มอีก → OpenAI
  fallbacks:
    - pronounce: ["gemini-flash-lite", "openai-gpt-4o-mini"]
  # cache prompt ซ้ำๆ ของบทเรียนเดียวกัน
  cache:
    type: "redis"
    redis_host: "localhost"

2. Serving layer (ตอน self-host): vLLM

vLLM คือ inference server — โปรแกรมที่รัน LLM model file (เช่น .safetensors, .gguf) แล้ว expose เป็น HTTP API ที่หน้าตาเหมือน OpenAI. ทำงานบน GPU.

ฟีเจอร์เด่นของ vLLM:

  • Continuous batching — รวม request หลายๆ ตัวที่เข้ามาพร้อมกันเป็น batch เดียว (ใช้ GPU เต็มประสิทธิภาพ)
  • PagedAttention — เทคนิคจัดการ memory แบบใหม่ที่ลดการใช้ RAM ของ KV-cache ลงเยอะ ทำให้รับ concurrent users ได้มากขึ้น
  • รองรับ quantization native — AWQ, GPTQ, FP8 ใส่เข้าไปใช้ได้เลย
  • คำสั่งเดียวก็ขึ้น: vllm serve scb10x/typhoon-v2.1-gemma3-12b

ทางเลือกอื่น:

  • TGI (Text Generation Inference จาก HuggingFace) — ใช้ได้ดีใกล้เคียง vLLM แต่ตามหลังเล็กน้อยใน architecture ใหม่ๆ (เช่น Qwen3-MoE)
  • Ollama / llama.cpp — ดีสำหรับ dev บน laptop หรือเครื่องตัวเอง. แต่อย่าเอามา serve production — ไม่มี continuous batching จริงๆ จะรับ concurrent users ได้น้อยมาก
  • หมายเหตุ: ถ้าเลือกใช้ไฟล์โมเดลแบบ GGUF (พบบ่อยในชุมชน open) — อย่ารันผ่าน vLLM เพราะ performance แย่ (~93 tok/s). ใช้ llama.cpp ตรงๆ ดีกว่าสำหรับ GGUF

3. Quantization (บีบโมเดลให้เล็กลง)

Quantization = การลดความแม่นของตัวเลข weights ในโมเดล (เช่น จาก float32 → int4) เพื่อให้ขนาดเล็กลงและรันเร็วขึ้น. ราคาที่จ่ายคือคุณภาพลดเล็กน้อย.

เลือกอันไหนดี:

  • Q4_K_M (GGUF) หรือ AWQ-INT4 (safetensors) — จุดที่คุ้มที่สุด. โมเดลเล็กลง ~75%, ความเร็วเพิ่ม ~2-3 เท่า, คุณภาพลดน้อยมาก (perplexity เพิ่ม ~+0.05 บน Llama-3.1-8B)
  • FP8 (vLLM) — ดีกว่า Q4 ถ้า GPU รองรับ (Hopper, Ada Lovelace). ความเร็ว ~2× ของ FP16, คุณภาพแทบไม่ลด
  • หลีกเลี่ยง Q3 หรือต่ำกว่าสำหรับภาษาไทย — tokenizer ภาษาไทยอัดข้อมูลหนาแน่นกว่าภาษาอังกฤษเยอะ (1 token ของไทย = หลายตัวอักษร), ฉะนั้นเมื่อ precision ลด ความเสียหายต่อภาษาไทยจะหนักกว่าภาษาอังกฤษ

แผน rollout แบบแบ่งเฟส (default สำหรับ SME)

หลักการ: ลงทุนต่ำสุดก่อน เห็นผลจริงค่อยขยับ. ทุกเฟสไม่ต้องแก้โค้ดแอปเลย — เพราะ LiteLLM คั่นกลางไว้แล้ว.

สำหรับ enterprise/government deployment (ต้อง commit capacity ตั้งแต่วันแรก, ข้อมูลถูกกำกับโดยกฎหมาย, ผู้ใช้หลายพัน) — ดู Enterprise rollout แทน — ลำดับต่างกัน (term sheet + DPA ก่อน provisioning)

เฟส 1 · สัปดาห์ 1

วาง LiteLLM gateway (ไม่ต้องใช้ GPU)

รัน LiteLLM ใน Docker. ตั้งค่า Primary = opentyphoon.ai, Fallback = Gemini Flash-Lite, สำรองอีกชั้น = OpenAI. เปิด semantic cache เพื่อลด token ซ้ำซ้อน. ผลที่ได้: rate-limit หาย, ค่าใช้จ่ายลด ~50%, ถ้า provider ใดล่ม ระบบจะ failover อัตโนมัติ. ใช้เวลา ~1 วัน. วัดความสำเร็จ: p95 latency ไม่เพิ่ม + ไม่มี request fail ในช่วง stress test

เฟส 2 · สัปดาห์ 2

Benchmark prompt จริงใน production

นำ prompt จาก GeminiPronunciationService.cs มารันผ่าน Typhoon 4B / 12B / 30B-A3B เทียบกับ Gemini. วัด 4 อย่าง: (1) schema compliance — โมเดลตอบ JSON ถูก format ทุกครั้งไหม, (2) latency p50/p95, (3) ความเป็นธรรมชาติของภาษาไทย (ให้คนตรวจมือ 50 ตัวอย่าง), (4) ค่าใช้จ่ายต่อ 1k requests. เลือกตัวที่ชนะตาม use case แต่ละแบบ. วัดความสำเร็จ: มีข้อมูลพอที่จะตัดสินใจ

เฟส 3 · เมื่อคุ้มค่า

Self-host Typhoon-12B

เกิดเมื่อ traffic แตะ ~50k req/วันแบบต่อเนื่อง 2-3 สัปดาห์ติด. เช่า GPU L40 จาก Vast.ai ($220/เดือน) รัน Typhoon 2.1-Gemma3-12B แบบ Q4_K_M ผ่าน vLLM. ใน LiteLLM แค่เพิ่ม model entry ใหม่ + เปลี่ยน Primary มาเป็น endpoint ตัวเอง — โค้ดแอปไม่ต้องแก้แม้แต่บรรทัดเดียว. วัดความสำเร็จ: ค่าใช้จ่ายต่อเดือนลดเทียบกับ hosted, latency ไม่เพิ่ม

เฟส 4 · เลือกได้ (ทำเมื่อพร้อม)

Whisper สำหรับโหมด audio

ติดตั้ง Whisper-large-v3 บน GPU ตัวเดียวกัน (ใช้ CTranslate2 หรือ whisper.cpp ให้เบาขึ้น). Pipeline ใหม่: เสียง → Whisper STT → transcript → Typhoon → JSON. หรือ ทางเลือก B: เก็บโหมด audio ไว้ที่ OpenAI ถาวรเป็น hybrid ก็ได้ (ลด complexity). วัดความสำเร็จ: คุณภาพการให้คะแนนการออกเสียงไม่ตกเกิน 5%

จุดที่ต้องระวัง

ไม่มี Thai LLM ตัวไหนรับ audio ได้

ปัญหา: โหมด audio ของแอปปัจจุบันใช้ gpt-4o-audio-preview ที่รับเสียงดิบเข้าไปประเมินได้เลย. แต่ยังไม่มี open Thai LLM ตัวไหนรับ audio เป็น input ได้ตรงๆ.

วิธีแก้:

  • ใส่ Whisper-large-v3 (open-source STT จาก OpenAI) เป็น preprocessor — แปลงเสียงเป็น transcript ก่อน, แล้วค่อยส่งให้ Thai LLM ประเมิน
  • หรือทำ hybrid — text-mode ใช้ Thai LLM, audio-mode คงไว้ที่ OpenAI ตลอดไป (ง่ายสุด)
Quantization สำหรับภาษาไทยอ่อนไหวกว่าภาษาอังกฤษ

ปัญหา: tokenizer ภาษาไทยใช้ token น้อยกว่าภาษาอังกฤษในการแทนข้อความขนาดเดียวกัน — แปลว่าแต่ละ token "อัด" ความหมายไว้แน่นกว่า. ฉะนั้นเมื่อ quantize ลด precision ของ weights, ผลกระทบต่อภาษาไทยจะรุนแรงกว่าภาษาอังกฤษ.

คำแนะนำ: ใช้ Q4_K_M เป็นขั้นต่ำสุด. FP8 ดีกว่ามากถ้า GPU รองรับ. อย่าใช้ Q3 หรือต่ำกว่ากับงานภาษาไทย — คุณภาพจะตกเห็นได้ชัด

Function calling เชื่อถือได้แค่ไหน

ปัญหา: งาน Enggenius ต้อง parse JSON ที่ LLM ตอบ. ถ้า LLM ตอบ JSON ผิด format (เช่น ลืม field, ใส่ comma เกิน, เพิ่มข้อความก่อน/หลัง JSON) แอปจะ crash.

มีแค่ Typhoon ที่ตีพิมพ์ตัวเลข BFCL (Berkeley Function Calling Leaderboard) — 75% TH / 79% EN. OpenThaiGPT และ Pathumma รองรับได้ตามที่ inherit จาก Qwen2.5 แต่ยังไม่มีใคร eval ด้วย prompt ภาษาไทยอย่างเป็นทางการ.

คำแนะนำ: ไม่ว่าจะเลือกเจ้าไหน — เขียน retry + repair logic ไว้เสมอ: ถ้า parse JSON fail ให้ retry, ถ้ายัง fail อีกให้ส่งไป fallback provider

Code-switching (ไทยปนอังกฤษ)

ปัญหา: นักเรียนไทยจริงๆ พูดภาษาผสม — เช่น "ฉันชอบ pizza มากกว่า burger", "เปิด tab ใน browser", "ฉัน save งานแล้ว". LLM ที่ tune มาเฉพาะภาษาไทยล้วนๆ อาจสับสน หรือพยายามแปลคำอังกฤษกลับมาเป็นไทยจนผิดความหมาย.

เกณฑ์: Typhoon และ SEA-LION จัดการ code-switching ได้ดี (เพราะ train บน data ที่มี mixed language). OpenThaiGPT 1.5 บางครั้งแปลคำอังกฤษกลับเป็นไทยมากเกินไป.

คำแนะนำ: ทดสอบ prompt ของจริง (เช่น transcript จากนักเรียนจริงที่มี code-switching) ก่อนเลือก vendor — อย่าเชื่อ marketing

ประสิทธิภาพของ tokenizer (สำคัญต่อค่าใช้จ่าย)

ปัญหา: ราคา $/M tokens ของแต่ละ vendor ดูเท่ากัน แต่ ภาษาไทยใช้ token ต่างกันมหาศาลในแต่ละ tokenizer.

  • Tokenizer ที่ปรับมาสำหรับไทยโดยเฉพาะ (Typhoon SentencePiece) — ใช้ token น้อยกว่า ~2.6 เท่า เมื่อเทียบกับ vanilla GPT-4 / Llama tokenizer สำหรับข้อความไทยเดียวกัน
  • Tokenizer ทั่วไป (Llama 3.1, Qwen2.5) — กินจำนวน token เพิ่ม ~2 เท่า สำหรับเนื้อหาไทยขนาดเดียวกัน

คำแนะนำ: เทียบราคา $/M tokens ตรงๆ ไม่พอ — ต้องลองยิง prompt ของจริงดูว่าแต่ละ vendor ใช้ token เท่าไหร่. โมเดลที่ราคาดูแพงกว่าอาจถูกกว่าจริงๆ ถ้า tokenizer ของมันประหยัดกว่า

แต่ละโมเดลมีเพดานเชิงพาณิชย์ต่างกันตาม base ที่มันมาจาก:

  • Llama-based (Typhoon 2-70B, SEA-LION-70B) — ติดเพดาน 700M MAU. ถ้าผู้ใช้ active เกิน 700 ล้านคน/เดือน ต้องขออนุญาต Meta ก่อน
  • Gemma-based (Typhoon 2.1-Gemma3) — ใช้ Gemma Terms of Use. ใช้เชิงพาณิชย์ได้ปกติ ไม่มีเพดาน MAU แต่ห้ามใช้ละเมิด policy ของ Google
  • Qwen-based (OpenThaiGPT, THaLLE) — ติดเพดาน 100M MAU
  • Apache-2.0 (Pathumma, Qwen3 base — Typhoon 2.5 สืบทอดจาก Qwen3) — เสรีที่สุด ไม่มีเพดาน ไม่มีเงื่อนไขเชิงพาณิชย์

ที่ scale SME — ยังไม่มีตัวไหนติดเพดานอยู่แล้ว แต่ควรเลือกโดยรู้เงื่อนไขล่วงหน้า เผื่อโตเร็วในอนาคต

Safety alignment / การกรองเนื้อหา

ปัญหา: Thai LLM ทุกตัวผ่านกระบวนการ RLHF (Reinforcement Learning from Human Feedback — การสอน LLM ให้ตอบดีๆ ผ่าน feedback ของคน) เบากว่า Gemini หรือ Claude. ผลคือ:

  • ดี: ปฏิเสธคำถามแปลกๆ น้อยลง — ไม่หงุดหงิดผู้ใช้
  • ไม่ดี: ทนต่อ jailbreak (การหลอกให้ LLM พูดสิ่งที่ไม่ควร) ได้น้อยกว่า — เปราะกับ prompt injection

คำแนะนำ: สำหรับแอปการศึกษาที่อาจมีเด็กเป็นผู้ใช้, อย่าฝากความปลอดภัยไว้ที่ LLM อย่างเดียว. ซ้อน moderation filter ของตัวเองทับอีกชั้น — เช่น keyword filter, classifier ที่ตรวจ output ก่อนแสดง

การ Deploy ระดับ Enterprise / รัฐบาล

คำแนะนำในเอกสารนี้ออกแบบมาสำหรับ workload ขนาด SME (ปริมาณ 50k–500k req/วัน, ทีม product เดียว). พอขยับขึ้นไปเป็น government / education / enterprise scale — เช่น โรงเรียนหลายร้อยแห่ง, ผู้ใช้หลายหมื่น, peak 1,000+ CCU — สมการเปลี่ยนหมด.

เมื่อไหร่ที่คำแนะนำ "อย่าเพิ่ง self-host" จะกลับด้าน

ที่ volume SME, hosted Typhoon API ถูกกว่า self-host. แต่พอข้าม ~50k req/วัน แบบยั่งยืน ทุกวัน — math กลับ: GPU ที่ reserve ราคา fix เริ่มถูกกว่าจ่ายตาม token. ที่ 500k+ req/วัน, self-host (หรือ managed self-host ผ่าน Thai cloud provider) ถูกกว่า hosted API มหาศาล.

ทำไม Thai-region cloud ถึงดีกว่าทั้ง hosted API และ DIY self-host

สำหรับ government deployment, มี 3 ข้อจำกัดมารวมกันที่เปลี่ยนการเลือก vendor:

  1. Data sovereignty (อธิปไตยของข้อมูล). ข้อมูลนักเรียน/ประชาชน/สุขภาพ ไม่สามารถออกจากประเทศได้ ตาม PDPA + ข้อกำหนดของหน่วยงาน. ข้อนี้ตัด hosted API ที่อยู่อเมริกาออกหมด (opentyphoon.ai ใช้ได้, แต่ Gemini / Claude / OpenAI ใช้ไม่ได้).
  2. Procurement-friendly. RFP ของรัฐชอบ vendor ที่มี local entity, สัญญาภาษาไทย, ใบกำกับภาษีไทย.
  3. ภาระ ops. เขตการศึกษาไม่อยากรัน vLLM บน Vast.ai เอง. ต้องการเบอร์โทรไว้โทรหา.

ฉะนั้น Thai managed cloud provider (SIAM.AI, NTT Thailand, AWS Asia-Pacific via Thai partner, Bizthai ฯลฯ) มาอยู่ใน sweet spot — รับเรื่อง GPU ไว้ และเปิด OpenAI-compatible endpoint, โดยข้อมูลอยู่ในไทยทั้งหมด.

โครงสร้างราคาที่ต้องเจรจา (อย่ารับ flat-only)

Government workload เป็น burst-shaped: ใช้หนักช่วงเวลาเรียน, แทบไม่มีการใช้ในวันหยุด/เสาร์อาทิตย์. flat-reserved pricing คือเสียเงินเปล่า. ต้องเจรจา:

  • Reserved baseline — รับ steady-state load (เช่น 200 CCU)
  • Token-based หรือ per-CCU burst — สำหรับ peak และช่วงสอบ
  • Price-review clause ที่เดือน 6 — ปรับได้สูงสุด ±10% ถ้า GPU upstream ขยับ >15%
  • Annual minimum commit พร้อม penalty (vendor ต้องการอันนี้) — แต่ขอ credit roll-over ให้ capacity ที่ไม่ใช้สามารถยกไปเดือนถัดไปได้

Compliance stack สำหรับข้อมูลผู้เยาว์ / ข้อมูลประชาชน

สำคัญมากสำหรับงานรัฐ

ถ้าผู้ใช้คือผู้เยาว์ (นักเรียน), เสียงและ transcript ของผู้เยาว์ถือเป็น special-category data ภายใต้ PDPA Section 26. NDA + VPC ไม่พอ. ต้องมี:

  • Data Processing Agreement (DPA) เซ็นแยกจาก master service agreement
  • Breach notification SLA: 24-72 ชม., ไม่ใช่ "best effort"
  • Auto-delete policy สำหรับ raw audio (แนะนำ ≤ 30 วัน)
  • Audit log retention: 3 ปี สำหรับงานรัฐ, 1 ปี ขั้นต่ำสำหรับเอกชน
  • Sub-processor disclosure — ต้องเปิดเผย vendor downstream ทุกราย
  • Right to audit — ลูกค้า (หรือหน่วยกำกับ) เข้าตรวจ infrastructure ได้รายปี
  • No-training clause — vendor ห้ามใช้ข้อมูลเราไป fine-tune model ของเขาหรือของบุคคลที่สาม
  • Parental consent flow สำหรับผู้เยาว์ — สร้างใน app, log ไม่สามารถลบได้

Capacity guarantee — ระบุยังไงให้ใช้ได้จริง

เขียน "1,000 CCU peak" ในสัญญาไม่พอ. SLA จริงต้องมี:

  • Reserved CCU: floor ที่ใช้ได้ตลอด (ไม่ shared)
  • Burst CCU: เพดานสูงสุด, มีนโยบายเมื่อ overflow (queue / soft throttle / fail)
  • TTFT p95 (time to first token) ที่ reserved load และที่ peak
  • End-to-end response time p95
  • Token throughput floor (tokens/วินาที) ที่ peak — แยกจาก response time
  • Overflow policy — ที่ 110% ของ burst CCU ทำยังไง?

4 phases สำหรับ enterprise rollout

Phase 1 · สัปดาห์ 1-2

Term sheet + procurement

Lock โมเดลราคา (hybrid), เป้า SLA, ตัวเลข capacity, scope ของ DPA. จัดการ procurement / legal / data protection officer ภายในให้พร้อม. อย่าข้ามขั้นนี้ — deploy โดยไม่มี DPA = ความเสี่ยงทางคดี โดยเฉพาะเมื่อข้อมูลผู้เยาว์เกี่ยวข้อง.

Phase 2 · สัปดาห์ 3-4

Provisioning + integration

Vendor reserve GPU pool. Integrate ผ่าน OpenAI-compatible endpoint (LiteLLM ยังมีประโยชน์เป็น fallback แม้จะใช้ managed provider). Acceptance test: load test ที่ 1,000 CCU, latency p95 ≤ 2s, schema compliance > 99%, ตรวจ Thai naturalness ที่ sample 100 prompts.

Phase 3 · สัปดาห์ 5-6

Soft launch (pilot subset)

Roll out ที่ ~10% ของ scope (เช่น 40-50 โรงเรียน). monitor SLA, รับ feedback, ปรับ Wav2Vec2 / model. แก้ปัญหาก่อน deploy เต็ม. อย่าข้าม soft launch — 437 โรงเรียน live พร้อมกันวันแรก = ขอ viral incident เลย.

Phase 4 · สัปดาห์ 7+

Full rollout + monitoring

Scale ทุกโรงเรียน. Vendor scale reserved capacity. ตั้ง monthly SLA review, weekly latency/cost dashboard, quarterly capacity planning. วางแผน renew negotiation 90 วันก่อนสัญญาหมด.

สรุป

สำหรับ SME / single-product: ใช้ hosted Typhoon API + LiteLLM gateway, ยังไม่ต้อง self-host จนกว่าจะพิสูจน์ volume.

สำหรับ enterprise / government: เลือก Thai managed cloud provider, เจรจา hybrid pricing, เซ็น DPA จริงๆ, roll out เป็น phases. คำเคลม saving 87% ที่เห็นใน vendor pitch สมมติว่า load เป็น burst-shaped — verify ใน contract terms ว่าสมมติฐานถูกต้อง.
