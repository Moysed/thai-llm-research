Replacing Gemini & GPT in a Thai English-learning app

A pragmatic 2026 review of open-source Thai LLMs — Typhoon, OpenThaiGPT, Pathumma, SEA-LION — for a real production workload (pronunciation grading + question generation), with self-hosting economics, deployment stack, and a phased rollout plan.

TL;DR

Reading guide

The sections below frame the decision for SME / single-product workloads (50k–500k req/day). If you're at enterprise or government scale (~1,000+ CCU, multi-thousand users, regulated data), the recommendations differ — jump to Enterprise & Government Deployment.

Note on audio mode: for SME, hybrid (text → Thai LLM, audio → OpenAI) is acceptable. For regulated/government data, audio must move to local Whisper + Wav2Vec2 — sending children's voices to US endpoints isn't compatible with PDPA.

Verdict (SME default)

Don't fully self-host yet. Drop a LiteLLM gateway in front of opentyphoon.ai with Gemini as fallback — eliminates rate-limit pain at near-zero ops cost. Self-host Typhoon-12B only once sustained traffic crosses ~50k calls/day.

The problem

An English-learning app for Thai students currently routes pronunciation feedback to Gemini 2.5 Flash-Lite and GPT-4o-audio-preview. Three risks:

  1. Vendor lock-in. If Anthropic / Google / OpenAI degrade, deprecate, or rate-limit your tier, the product breaks.
  2. Token costs scale linearly with users. At low volume the hosted APIs are cheap; at high volume the ratio inverts.
  3. Sovereignty. Student speech recordings flow through US-hosted endpoints. Some institutional buyers care.

Goal: a migration path to a Thai-hosted LLM (whether self-hosted, hosted via opentyphoon.ai, or via a managed Thai cloud provider) that doesn't break the product on day 1. The right answer depends on scale — see the reading guide above.

The actual workload (don't skip this)

Picking an LLM without measuring the workload is theater. The current calls look like this:

  • Prompt size: ~500 input tokens (system + reference phrase + student transcript).
  • Output: 200–400 tokens of structured JSON — word-level scores 0-100, grammar errors, bilingual EN/TH feedback, example sentences.
  • Latency: real-time, user-facing. p95 budget <5s.
  • Volume: estimated 500–10,000 calls/day depending on active students.
  • Two modes: text-mode (transcript reasoning) and audio-mode (raw audio in). Audio mode is what no open Thai LLM can do natively today.
  • No function calling; the app uses response_schema / response_format: json_object.
Key constraint

If you keep audio-mode evaluation (raw audio → score), you can't 1:1 replace it with a text-only Thai LLM. You need a Whisper STT pre-processor, or you accept some quality loss by switching to text-mode only. Hybrid (text → Thai LLM, audio → OpenAI) is a perfectly fine middle path.

Top 3 candidates

Note: rankings below assume SME workload (hosted API or modest self-host). At enterprise/gov scale, the access path may shift to a managed Thai cloud provider (e.g. SIAM.AI, NTT Thailand) — same models, different deployment vehicle. See Enterprise & Government for that decision tree.

1Typhoon 2.x — SCB-10X

Best Thai naturalness Function calling Hosted API Apache-2.0 (2.5)

Sizes: 4B (Gemma3 base), 12B (Gemma3, 128K ctx), 30B-A3B (Qwen3 MoE — ~3B active), 70B (Llama 3.1).

The only Thai LLM with published BFCL function-calling numbers (75% TH / 79% EN — beats Qwen2.5/Llama3 baselines). Hosted at opentyphoon.ai at ~$0.10/M tokens (~14× cheaper than Gemini Flash 2.5 per their 2.5 release post). #1 across every independent Thai leaderboard.

Why this fits (SME deployment): 4B fits a single L40 GPU at production load, JSON schema compliance is reliable, Thai feedback reads naturally, and the hosted API exists as a zero-code fallback. No surprises. For enterprise/government deployment, the same models are accessed via a managed Thai cloud — see below.

2OpenThaiGPT — NECTEC / AIEAT

Strong reasoning Qwen license (100M MAU cap) Free playground

Latest: 1.6 + R1-32B (Mar 2025). The R1-32B reasoning variant scores AIME24-TH 56.7 and MATH500-TH 83.8 — beating DeepSeek-R1-70B at half the size.

Inherits Qwen2.5 tool-calling. 131K input / 8K output. Free playground at openthaigpt.aieat.or.th but no enterprise SLA.

Why this fits: If you ever need long-context reasoning (multi-lesson conversations, complex grammar coaching), the R1 variant is the reasoning ceiling. Slower release cadence than SCB-10X — typical of state-funded research projects.

3Pathumma-LLM — NECTEC

Apache-2.0 Mid-tier quality Government-backed

Pathumma-llm-text-1.0.0 (Qwen2.5-7B base, 7B params, 128K ctx). Vision-2.0.0-preview and Audio-1.0.0 also out.

Quality trails Typhoon and OpenThaiGPT 1.5-14B. Less independently benchmarked. Hosted free at aiforthai.in.th.

Why this fits: The only zero-restriction license in the list. Keep as Plan B if Gemma ToU or Qwen 100M MAU clauses ever block enterprise procurement. Quality penalty is real but acceptable.

Also-rans (worth knowing about)

  • SEA-LION v3 / v3.5 (AI Singapore) — best when you also need Bahasa / Vietnamese / Tamil. For Thai-only, Typhoon wins.
  • WangchanLion / WangchanX (VISTEC) — more a fine-tuning pipeline than a model line. Strong for Thai MRC, weaker for chat.
  • THaLLE (KBTG) — finance-specialized. Overkill for an education app.
  • Original SeaLLM-v3 (Alibaba DAMO) — now legacy; AI Singapore's SEA-LION line is the live successor.

Reality check: at SME volume, self-hosting is not cheaper than hosted Gemini Flash-Lite. The win is rate-limit elimination and sovereignty.

Thai managed cloud = SIAM.AI, NTT Thailand, Bizthai etc. Higher than DIY self-host because of managed-service overhead, but the only viable option when sovereignty + procurement-friendliness matter.

GPU rental tiers

  • Cheapest: Vast.ai L40 40GB ~$0.31/hr (~$220/mo) — fits Typhoon 2.1-Gemma3-12B Q4_K_M comfortably.
  • Mid: RunPod A100 80GB ~$0.79/hr ($350–570/mo on-demand vs reserved) — runs 12B FP16 with KV-cache headroom.
  • Production: H100 80GB ~$1.50/hr ($1,100/mo) — only justified at >200 RPS. Typhoon 2.5-30B-A3B claims 3,000 tok/s here.

Production stack

Routing layer: LiteLLM

LiteLLM is the right default — self-hosted Docker, free, OpenAI-compatible. One YAML config gets you fallback chains, retry/cooldown, budget caps, and semantic caching. OpenRouter is hosted (zero ops, less control); Portkey is enterprise-grade (overkill for SME).

# litellm-config.yaml — example
model_list:
  - model_name: pronounce
    litellm_params:
      model: openai/typhoon-v2.1-12b-instruct
      api_base: https://api.opentyphoon.ai/v1
      api_key: os.environ/TYPHOON_API_KEY

router_settings:
  fallbacks:
    - pronounce: ["gemini-flash-lite", "openai-gpt-4o-mini"]
  cache:
    type: "redis"        # semantic cache for repeated lesson prompts
    redis_host: "localhost"

Serving (when self-hosting): vLLM

vLLM serves the OpenAI-compatible API, has continuous batching, paged attention, native AWQ/GPTQ/FP8. For a Qwen/Llama/Gemma-based Thai model, vllm serve is one command.

  • TGI (HuggingFace) is comparable but lags vLLM on new architectures.
  • Ollama / llama.cpp are fine for dev and laptop benchmarks. GGUF served from vLLM is poor (~93 tok/s); use llama.cpp directly if you go GGUF.
  • Don't serve Ollama in production — no proper continuous batching.

Quantization

Q4_K_M GGUF or AWQ-INT4 is the sweet spot — Q4_K_M adds only ~+0.05 perplexity on Llama-3.1-8B and the imatrix variants preserve rare-token quality. Avoid Q3 and below for Thai — Thai's denser tokenizer footprint makes precision loss propagate further than in English. FP8 on vLLM gives ~2× throughput vs FP16 with negligible quality loss.

Phased rollout (SME default)

For enterprise/government deployments (committed capacity from day 1, regulated data, multi-thousand users), see the Enterprise rollout instead — different sequencing (term sheet + DPA before provisioning).

Phase 1 · Week 1

LiteLLM gateway, no GPU

Drop in LiteLLM Docker. Primary: opentyphoon.ai. Fallback: Gemini Flash-Lite, then OpenAI. Add semantic cache. Rate-limit pain dies. Vendor-independent because LiteLLM swaps providers via YAML.

Phase 2 · Week 2

Real-prompt benchmark

Run actual production prompts through Typhoon 4B / 12B / 30B-A3B vs Gemini. Measure schema compliance, latency p50/p95, Thai naturalness (manual grade), cost. Pick the winner per task.

Phase 3 · When justified

Self-host Typhoon-12B

When sustained traffic crosses ~50k req/day, spin up Vast.ai L40 ($220/mo) running Typhoon 2.1-Gemma3-12B Q4_K_M via vLLM. LiteLLM swaps endpoint — zero app code change.

Phase 4 · Optional

Whisper for audio mode

Whisper-large-v3 self-hosted on the same GPU (CTranslate2 or whisper.cpp) → transcript → Typhoon → JSON. Or keep audio routed to OpenAI as permanent hybrid. Both are fine.

Gotchas that will bite you

No audio-in

No open Thai LLM has audio-in capability. If you must keep audio-mode evaluation, plan a Whisper preprocessing step or stay hybrid with OpenAI for audio.

Quantization sensitivity

Q3 and lower quantization breaks Thai harder than English. Thai's tokenizer maps characters to fewer denser tokens, so each lost bit of precision propagates further. Q4_K_M minimum; FP8 ideal.

Function-calling reliability

Only Typhoon publishes BFCL-style numbers (75% TH / 79% EN). OpenThaiGPT and Pathumma technically support tool-calling via Qwen2.5 inheritance but lack independent Thai-prompt evals. Write retry + repair logic regardless of provider.

Code-switching

Real Thai input mixes Thai and English ("เปิด tab ใน browser"). Typhoon and SEA-LION handle this best; OpenThaiGPT 1.5 sometimes over-translates English back to Thai. Test your exact prompts before committing.

Tokenizer efficiency

Thai-tuned tokenizers (Typhoon SentencePiece) are ~2.6× more efficient on Thai than vanilla GPT-4 / Llama tokenizers. Vanilla Llama 3.1 / Qwen2.5 burn ~2× the tokens for the same Thai text — so cost comparisons must factor tokenizer fit, not just $/M tokens.

License terms

Each model inherits its commercial ceiling from the base it was fine-tuned on:

  • Llama-based (Typhoon 2-70B, SEA-LION-70B) — 700M MAU clause. Past 700M monthly active users you need Meta's permission.
  • Gemma-based (Typhoon 2.1-Gemma3) — Gemma Terms of Use. Commercial use allowed, no MAU cap, but you must comply with Google's prohibited-uses policy.
  • Qwen-based (OpenThaiGPT, THaLLE) — 100M MAU cap.
  • Apache-2.0 (Pathumma, Qwen3 base — Typhoon 2.5 inherits this from Qwen3) — most permissive, no cap, no commercial restriction.

None of these block an SME today, but choose with the ceiling in mind — switching later if you grow fast is more painful than picking right up front.

Safety alignment

Thai LLMs are less aggressively RLHF'd than Gemini/Claude — fewer refusals, but weaker jailbreak resistance. For a kids-adjacent education app, layer your own moderation filter on top.

Enterprise & Government Deployment

The recommendations above target SME workloads (50k–500k req/day, single product team). At government / education / enterprise scale — think hundreds of schools, tens of thousands of users, peak 1,000+ CCU — the calculus shifts.

When the "don't self-host yet" advice flips

For SME volume, hosted Typhoon API is cheaper than self-host. Past ~50k req/day sustained, the math inverts: reserved GPU starts beating per-token billing. At 500k+ req/day, self-host (or managed self-host via a Thai cloud provider) is dramatically cheaper than any hosted API.

Why a Thai-region cloud beats both hosted API and DIY self-host

For government deployments, three constraints come together that change the vendor decision:

  1. Data sovereignty. Student / citizen / health data cannot leave the country under PDPA + agency regulations. This rules out US-based hosted APIs (opentyphoon.ai is OK, but Gemini / Claude / OpenAI are not).
  2. Procurement-friendliness. Government RFPs favor in-country vendors with local entities, Thai-language contracts, Thai-region invoicing.
  3. Operational burden. A school district doesn't want to run vLLM on Vast.ai. They want a phone number to call.

That puts managed Thai cloud providers (SIAM.AI, NTT Thailand, AWS Asia-Pacific via local partner, Bizthai, etc.) in the sweet spot — they handle the GPU and provide an OpenAI-compatible endpoint, while keeping data inside Thailand.

Pricing structure to negotiate (not flat-only)

Government workloads are burst-shaped: heavy during school hours, near zero on weekends and holidays. Flat-reserved pricing wastes budget. Always negotiate:

  • Reserved baseline — covers steady state (e.g. 200 CCU)
  • Token-based or per-CCU burst — for peaks and exam weeks
  • Price-review clause at month 6 — max ±10% if upstream GPU prices move >15%
  • Annual minimum commit with under-use penalty (vendor wants this) — but with credit roll-over so unused capacity isn't lost

Compliance stack for minors' / citizen data

Critical for government

If your users are minors (school students), their voice and transcript data is special-category data under PDPA Section 26. NDA + VPC alone is not sufficient. Required:

  • Signed Data Processing Agreement (DPA) separate from the master service agreement
  • Breach notification SLA: 24-72 hours, not "best effort"
  • Auto-delete policy for raw audio (recommend ≤ 30 days)
  • Audit log retention: 3 years for government, 1 year for private sector minimum
  • Sub-processor disclosure — every downstream vendor must be listed
  • Right to audit — customer (or their regulator) can inspect infrastructure annually
  • No-training clause — vendor cannot use your data to fine-tune their models or any third-party model
  • Parental consent flow for minors — built into the app, logged immutably

Capacity guarantee — what to actually specify

Writing "1,000 CCU peak" in a contract isn't enough. A real SLA needs:

  • Reserved CCU: a floor that's always available (not shared)
  • Burst CCU: an upper ceiling, with defined behavior past it (queue / soft throttle / fail)
  • TTFT p95 (time to first token) at reserved load AND at peak
  • End-to-end response time p95
  • Token throughput floor (tokens/sec) at peak — separately from response time
  • Overflow policy — what happens at 110% of burst CCU?

The 4-phase rollout (revised for enterprise)

Phase 1 · Week 1-2

Term sheet + procurement

Lock pricing model (hybrid), SLA targets, capacity numbers, DPA scope. Get internal procurement / legal / data protection officer aligned. Don't skip this — deploying without a DPA is a litigation risk when minors' data is involved.

Phase 2 · Week 3-4

Provisioning + integration

Vendor reserves GPU pool. Integrate via OpenAI-compatible endpoint (LiteLLM still useful for fallback even with managed provider). Acceptance tests: load test at 1,000 CCU, latency p95 ≤ 2s, schema compliance > 99%, Thai naturalness check on 100 sample prompts.

Phase 3 · Week 5-6

Soft launch (pilot subset)

Roll out to ~10% of footprint (e.g. 40-50 schools). Monitor SLA, gather user feedback, tune Wav2Vec2 / models. Address issues before full deploy. Don't skip soft launch — 437 schools all going live on day 1 is asking for a viral incident.

Phase 4 · Week 7+

Full rollout + monitoring

Scale to all schools. Vendor scales reserved capacity to match. Set up monthly SLA reviews. Weekly latency / cost dashboard. Quarterly capacity planning. Plan for renewal negotiation 90 days before contract end.

Bottom line

For SME / single-product workloads: stick with hosted Typhoon API + LiteLLM gateway, defer self-hosting until proven volume.

For enterprise / government workloads: pick a Thai managed cloud provider, negotiate hybrid pricing, sign a real DPA, and roll out in phases. The 87% cost-saving claims you'll see in vendor pitches assume burst-shaped load — verify the assumption holds in your contract terms.
